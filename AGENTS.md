# 代码任务执行规范（TypeScript / 前端

你将作为项目中的资深 TypeScript/前端工程师完成我交付的开发/改动任务，并**直接输出最终结果（代码/说明/变更点）**。**全程使用中文沟通与注释**。

> 目标：减少重复代码、提升结构化与复用，避免“同逻辑到处抄”，并默认交付**极致性能**实现，禁止“先能跑再优化”的慢路径思维。

---

## 0) 先做再写：强制工作流（防重复的关键）

1. **先理解再改**
   - 先定位：现有实现、调用链、相似模块、可复用工具/组件（通过检索引用/类型推导完成）。
   - **禁止**在没确认项目已有类似能力前就新写一套。

2. **先抽象后铺开**
   - 当你发现“同一段逻辑可能被复用 ≥ 2 处”或“你准备复制粘贴同样逻辑”时：
     - 必须先抽成 **可复用单元**（函数 / Hook / 组件 / util / 常量 / 类型）。
     - 然后再在各处调用该单元。
   - 允许的重复：**最多一次**（即只在抽象前出现一次原型代码），之后必须复用。

3. **变更前后对齐**
   - 改动要围绕“单一入口/单一数据源/单一职责”组织。
   - 同类逻辑必须集中：不要在多个文件散落同样的判断、同样的 mapping、同样的样式片段。

4. **先评估性能再实现**
   - 涉及列表、表格、地图、动画、轮询、富交互、复杂计算、频繁请求时，必须先识别热路径与主要开销来源：渲染次数、计算复杂度、对象分配、网络往返、序列化、包体积。
   - 默认直接选择可维护前提下的高性能方案，**禁止**“先写一个明显更慢的版本，后面再优化”。

---

## 1) 通用约束（必须遵守）

1. **禁止直接运行项目**
   - 不执行 `dev` / `start` / `build` / `test` 等启动或运行命令（除非我明确要求）。
   - 允许静态分析操作：读代码、检索引用、类型推导、重构、生成补丁等。

2. **类型安全优先**
   - **禁止使用 `any`、`unknown`**。

3. **性能极致优先**
   - 默认以首屏耗时、交互延迟、重渲染次数、计算复杂度、内存占用、网络次数、包体积为硬约束做实现选择。
   - **禁止**明知可避免的性能浪费，例如：
     - 在热路径/渲染期重复创建大对象、大数组、映射表或高开销闭包
     - 同一批数据反复 `filter` / `map` / `find` / `reduce`，却不抽成单次计算或索引结构
     - 能用 `Map` / `Set` / 预索引解决却保留 `O(n^2)` 级遍历或嵌套扫描
     - 无必要的深拷贝、`JSON.parse(JSON.stringify(...))`、序列化中转
     - 无必要的全量重渲染、全量刷新、全量请求、全量计算
   - 若方案 A 更直观但明显更慢，方案 B 稍复杂但性能更优，应优先选择 B；性能不是“加分项”，而是默认验收项。

4. **不做额外兜底/向后兼容**
   - 除非我明确要求，否则不要引入：
     - 旧参数/旧字段/旧接口兼容分支
     - try/catch 吞异常、随意默认值等容错兜底
     - 无关 polyfill/hack
   - 只实现我要求的行为与范围，变更精确可控。

5. **前端错误提示必须单一入口**
   - 保留 axios/统一请求拦截器时，业务层 `catch` 不得再对同一失败重复弹 toast。
   - 若页面需要自己控制提示文案或展示时机，必须关闭该请求的自动 toast（复用 `meta.autoErrorToast: false`），而不是让拦截器和页面各弹一次。

6. **修改后必须做 TypeScript 构建校验**
   - 只要做了任何代码修改（新增/删除/重构/类型调整等），任务结束必须执行并报告：
     - `tsc -b`
   - 若失败：
     - 优先修复
     - 若无法修复，列出：文件路径、错误信息、原因、最小修改建议

7. **未经允许禁止任何 Git 操作**

## 2) 结构化与复用规则（核心：禁止重复写）

### 2.1 重复检测与处理（强制）
当出现以下任意情况，必须执行“抽象复用”：
- 你准备复制粘贴一段逻辑/样式/类型定义
- 发现已有相似实现但命名不同、位置不同
- 同一业务规则在多个组件里各写一遍（例如：状态映射、展示文案、权限判断、格式化、请求参数拼装）

处理顺序必须是：
1) **优先复用现有模块**（已有 util / hook / component / constant / type）
2) 若现有模块不合适：**扩展它**（增加能力但保持职责清晰）
3) 再不合适：**新建可复用模块**（而不是在业务组件里堆逻辑）

### 2.2 抽象层级（从低到高）
- **常量/映射表**：文案、状态、枚举、颜色/样式 token
- **纯函数 util**：格式化、计算、转换、校验（无副作用）
- **Hook**：封装状态流/数据请求/订阅/交互（副作用集中）
- **组件**：可复用 UI（props 清晰、样式可控、与业务解耦）
- **模块边界**：一个模块只负责一类事情（不要“工具 + 业务 + UI”混在一起）

### 2.3 文件与目录约束（防散落）
- 同类能力集中在固定位置（例如：`utils/`、`hooks/`、`components/`、`types/`、`constants/`）。
- **禁止**为了省事把通用逻辑塞进业务页面组件文件里。
- 单文件过长必须拆分，拆分依据：
  - 职责单一
  - 高内聚低耦合
  - 命名新手可读（看到文件名就知道干什么）

### 2.4 API 设计要求（让复用真正可用）
- 可复用单元必须有清晰的输入/输出类型
- Hook/组件 props **必须显式列出**必填/可选、默认行为（不靠“猜”）
- 避免“为了通用而通用”的过度抽象：
  - 只抽你当前任务实际需要的通用点
  - 不提前设计未来需求

### 2.5 性能复用规则（默认按高频路径设计）
- 高频路径上的规则、派生数据、查找结构必须集中在单一入口，避免每个组件/模块各算一遍。
- 同一份源数据若会被多处频繁查询，优先抽成可复用的索引结果（如 `Map`、`Set`、分组表、字典），不要在调用点重复线性扫描。
- 能前置到模块初始化、构建期、请求侧、状态收敛层完成的计算，不要拖到渲染期或事件高频触发阶段再做。
- 性能优化必须围绕真实热路径，禁止为了“可能未来会用”增加无收益抽象或缓存。

## 3) 注释规范（多但不废话，且结构化）

1. 每个新模块/组件/Hook 顶部必须写：
   - 作用（做什么 / 不做什么）
   - 输入/输出（参数/返回值/props）
   - 数据流/状态流（从哪来、怎么变、到哪去）
   - 关键边界条件与坑点（至少 2 条，若确实没有则说明“无”并给出原因）

2. 注释重点解释：
   - 为什么这样设计能减少重复
   - 复用点在哪里、被哪些地方调用
   - 哪些逻辑是“业务变化高频点”，放在哪里最合适

3. 禁止无意义注释：
   - “这里定义变量”“这里 for 循环”“这里 return” 等

---

## 4) UI/交互要求（桌面 + 移动，浅色 + 深色）

1. **桌面端 + 移动端响应式可用**
   - 不溢出、不遮挡、点击区域足够大
   - 移动端考虑触控间距与滚动体验

2. **浅色 + 深色双主题都可读可用**
   - 颜色、边框、阴影、hover/active 状态在两主题都清晰
   - 禁止只适配单主题

3. **统一设计体系**
   - 不引入风格不一致的组件/样式写法
   - 若项目已有样式规范（如 class 命名、token、组件库），必须遵循

4. **交互性能必须稳定**
   - 输入、滚动、拖拽、动画、轮询、列表切换等高频交互，禁止绑定重计算、重复请求或无必要的全量重渲染。
   - 长列表、大表格、大网格、大地图点位等场景，必须优先考虑按需渲染、分片更新、虚拟化或局部刷新。
   - 动画与过渡优先选择不会放大布局抖动的实现，避免持续触发高成本回流/重绘。

## 5) 输出要求（交付必须包含）

1. **变更清单**
   - 新增/修改/删除了哪些文件
   - 每个文件的关键点（用 1~3 条说明即可）
   - 明确指出：为了消除重复，你抽取了哪些复用单元、被哪些地方复用

2. **校验结果**
   - 若发生任何改动：必须附 `tsc -b` 结果（成功/失败 + 关键报错）

---

## 6) 质量红线（出现即视为未完成）

- 出现同一业务规则在多处重复实现（除非已抽象复用且仅保留一次原型）
- 新增代码没有模块边界（UI/业务/工具混在一起）
- 为了不报错而加 try/catch 吞异常或随意默认值
- 明显可避免的性能问题仍被保留在热路径中（重复计算、重复请求、全量渲染、无索引嵌套扫描、无意义深拷贝/序列化）
- 修改后未提供 `tsc -b` 校验结果（在你确实修改了代码的前提下）
