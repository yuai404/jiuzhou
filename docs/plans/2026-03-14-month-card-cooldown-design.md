# 月卡冷却缩减 Design

## 背景

当前月卡只提供每日灵石奖励，没有参与其他玩法数值。伙伴招募与洞府研修都已经把“冷却计算”收敛为共享纯函数，服务层只消费计算结果。这意味着如果直接在两个 service 里分别判断“月卡是否激活”，会把同一条业务规则拆成两份，后续再加更多月卡效果时也会继续扩散。

## 目标

在修行月卡激活期间，为以下玩法提供 10% 冷却缩减：

- 伙伴招募
- 洞府研修

缩减后的冷却必须同时影响：

- 状态接口返回的剩余冷却
- 创建任务时的服务端拦截

## 方案

### 1. 抽取月卡冷却权益共享模块

新增一个服务端共享模块，负责两件事：

- 判断角色是否存在有效月卡
- 根据“基础冷却秒数 + 月卡激活状态”计算实际冷却秒数

这里不让伙伴招募和洞府研修自己维护“乘以 0.9”的逻辑，避免后续再出现第三个玩法时继续复制。

### 2. 冷却缩减口径

采用“基础冷却直接打 9 折”的方案：

- 伙伴招募：`168h -> 151.2h`
- 洞府研修：`72h -> 64.8h`

内部统一按秒计算，并继续使用“剩余秒数向上取整”的现有口径，保证倒计时展示与服务端拦截一致。

### 3. 接入位置

只改两个共享冷却入口，不改各自业务层判断结构：

- `buildPartnerRecruitCooldownState`
- `buildTechniqueResearchCooldownState`

服务层继续调用这两个共享函数，因此状态查询与创建拦截会自动同步生效。

## 边界

- 非生产环境现有“跳过冷却”规则保持不变；一旦跳过冷却，月卡缩减不再额外生效。
- 不新增向后兼容分支，不对其他玩法冷却做联动修改。
- 不要求前端新增月卡效果展示文案，本次只实现实际数值效果。

## 测试策略

先补纯函数测试覆盖两条关键行为：

- 未激活月卡时，冷却仍等于当前基础值
- 激活月卡时，冷却会缩短 10%，且结果同时影响 `cooldownUntil` 与 `cooldownRemainingSeconds`

最后运行 `tsc -b` 做全量 TypeScript 校验。
