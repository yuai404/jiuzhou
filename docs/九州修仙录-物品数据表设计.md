# 九州修仙录 - 物品数据表设计（字段策划）

> 版本：v0.1  
> 日期：2026-01-30  
> 目标：给出可覆盖全部玩法的“物品数据表”字段设计，支持后续落库到 PostgreSQL（表/字段中文备注齐全）。

## 1. 设计原则

- 数据分层：**物品定义（静态）** 与 **物品实例（动态）** 分离。
- 可扩展：对“效果、属性、词条、套装、配方、掉落、商店、任务”等采用扩展表或 JSONB 兼容。
- 可运营：支持版本控制、上下架、分服/分渠道、限时活动、热修配置。
- 可战斗：支持回合制技能、消耗、施放条件、命中/豁免、DOT、召唤、护盾等效果表达。
- 可交易：支持绑定、拍卖、交易税、价格区间、限购、回收价、禁交易。

## 2. 核心概念与必备表

为了覆盖“完整全部可能”，建议最少分为两层：

- 物品定义表：`item_def`（静态配置，策划可控）
- 物品实例表：`item_instance`（玩家背包/仓库/交易行中真实存在的物品）

其余为可选扩展表（当你不希望在 `item_def` 使用 JSONB 时启用）：

- `item_def_i18n`（多语言/多描述）
- `item_def_effect`（物品效果列表）
- `item_def_attr`（物品基础属性/成长属性）
- `item_def_affix_pool` / `item_instance_affix`（随机词条）
- `item_def_recipe` / `item_def_recipe_cost`（合成/锻造/炼丹配方）
- `item_def_drop_rule`（掉落规则）
- `shop_listing`（商店上架）
- `auction_listing`（拍卖行挂单）

- category：`consumable`（消耗品）/ `equipment`（装备）/ `material`（材料）/ `skillbook`（功法/秘籍）/ `quest`（任务）/ `other`（其他）

## 7. 装备系统（暗黑随机词条风格）

> 目标：掉落即生成“带随机词条的装备实例”，品质决定词条数量；装备的境界等级决定词条数值区间。

### 7.1 装备范围与槽位

- 装备属于 `category=equipment`。
- 槽位字段使用 `equip_slot` 表达（不绑定武器类型）：
  - `weapon`（武器）
  - `head`（头部）
  - `clothes`（衣服）
  - `gloves`（护手）
  - `pants`（下装）
  - `necklace`（项链）
  - `accessory`（饰品）
  - `artifact`（法宝）

说明：
- 武器不限定“剑/刀/枪”等类型，技能与功法不依赖武器类型；武器更多承担数值、词条与特效载体。

### 7.2 装备生成的关键概念

- 装备模板（静态）：在 `item_def` 中配置“这件装备可能长什么样”（名称、槽位、基础属性、可用词条池等）。
- 装备实例（动态）：在 `item_instance` 中保存“这次掉落出来的结果”（品质、词条条目与数值、强化/精炼、锁定等）。
- 装备境界等级（用于决定词条数值）：建议用 `equip_req_realm` + `level` 共同表达。
  - `equip_req_realm`：决定“最低可穿戴境界”
  - `level`：决定“同境界内的装备档位/强度段”

### 7.3 品质与词条数量规则（按你给的要求）

掉落生成实例时，根据品质决定词条数量：

- 黄：1～2 条
- 玄：2～4 条
- 地：4～5 条
- 天：6 条

建议补充规则（避免极端波动与刷不到关键词条）：
- 同一件装备的“词条条数”在掉落时一次性确定并固定在实例上。
- 天品 6 条中，允许出现“传奇词缀”（特殊词条）替换普通词条，但不额外增加条数。

### 7.4 词条结构（暗黑风格）

词条分为三层概念：**词条类型 → 档位（Tier）→ 具体数值Roll**。

- 词条类型（AffixKey）：例如 `wugong_flat`（物攻+）、`crit_rate`（暴击率+）、`heal_taken`（受治疗+）等
- 档位（Tier）：同一词条类型的“强度档”，由装备境界等级决定可出现的最高Tier
- 数值Roll：在该Tier的区间内随机，生成最终数值并写入实例

建议把词条按功能分组，便于做池子与权重：

1) 输出类：物攻/法攻、暴击/爆伤、易伤、对特定目标增伤（Boss/妖兽/人形）
2) 生存类：气血上限、物防/法防、护盾效率、受疗加成、减疗抵抗
3) 控制类：控制命中、控制抗性、控制持续时间+/-、净化效果增强
4) 资源类：灵气上限、回合回气、技能消耗降低、偷气/燃气相关
5) 功能类：金币/材料掉落加成、秘境掉落加成、战斗外收益（可选）

### 7.5 词条池（Pool）与约束（必做，否则会崩）

为了“完整考虑”，词条池需要支持：

- 槽位池：不同槽位有不同词条集合（例如武器更偏输出；衣服更偏生存）
- 品质池：天/地出现更多强词条与特殊词条
- 权重：每条词条有抽取权重（避免“全是垃圾词条”的体验）
- 互斥组：同组词条不可同时出现（例如 `wugong_flat` 与 `wugong_percent` 允许共存与否可配置；`crit_rate` 与 `anti_crit` 不应同时出现在同一件装备上，按需要）
- 上限组：某类词条在一件装备上总次数上限（例如“资源类最多2条”）

建议用 `affix_pool_id` 指向词条池配置；掉落时按如下顺序抽取：

1) 先决定词条条数（按品质）  
2) 从池子中按权重抽取词条类型（应用互斥/上限约束）  
3) 对每条词条决定Tier与数值Roll  
4) 写入 `item_instance.affixes`（或拆表 `item_instance_affix`）

### 7.6 装备境界等级如何影响词条数值（核心规则）

你提出“根据装备的境界等级决定词条数值”，推荐用“境界Rank + 装备Level档位”映射到可用Tier范围：

- 境界Rank：将境界体系映射为 1..13（凡人=1，成圣期=13），用于计算可出现Tier上限
- 装备Level：同境界内分段（例如 1..30），用于微调Tier上限与数值区间

示例规则（数值结构示意）：

- `maxTier = clamp(floor((realmRank + levelSegment) / 2), 1, 8)`
- 对于每个词条类型，预设：
  - `tierRange`：每Tier一段数值区间（例如物攻+：T1=1~2，T2=3~5，…）
  - `tierWeight`：越接近 maxTier，权重越低（避免全是顶级）
- 掉落时：
  - 根据 maxTier 抽取一个 `tier <= maxTier`
  - 在该tier区间内 Roll 数值

### 7.7 固有属性（Implicit）与随机词条（Explicit）

为了让装备“像暗黑”，建议拆成两部分：

- 固有属性（Implicit）：装备模板自带的固定属性（比如武器基础物攻、衣服基础气血/防御）
  - 表达方式：复用 `base_attrs` 或新增 `implicit_attrs`（JSONB）
- 随机词条（Explicit）：掉落时生成的随机属性（写入实例 `affixes`）

### 7.8 掉落生成流程（完整链路）

1) 确定掉落类型：装备 or 其他物品  
2) 确定装备槽位：按怪物/副本/宝箱掉落表配置  
3) 选择装备模板：从候选 `item_def` 中按权重抽取  
4) 生成品质：按掉落来源与保底规则决定（黄/玄/地/天）  
5) 生成装备境界等级：来源带入（怪物境界/副本推荐境界）+ 随机波动，落到 `equip_req_realm` 与 `level`  
6) 决定词条条数：黄1-2、玄2-4、地4-5、天6  
7) 抽取词条类型：按 `affix_pool_id` 权重与互斥规则抽取  
8) 生成Tier与数值：由装备境界等级决定Tier上限，最终Roll  
9) 写入实例：`item_instance`（含 `affixes`、`random_seed`、`locked=false` 等）  
10) 展示与拾取：进入背包（可选：未鉴定状态）

### 7.9 鉴定、重铸、分解、保底（完整玩法补齐）

鉴定（可选但强烈建议）：
- 掉落的地/天品装备可处于“未鉴定”状态：仅显示槽位与品质，不显示具体词条
- 鉴定消耗：灵石/鉴定符（材料），鉴定后生成/揭示词条

重铸（暗黑核心）：
- 重铸词条：消耗材料 + 灵石，对装备随机词条重新Roll（可锁定1条，锁定成本递增）
- 重铸不改变品质与词条条数，仅改变词条类型/Tier/数值（规则可配置）

分解与回收：
- 分解产出：词条相关材料（例如“玄晶粉”“道纹碎片”），用于重铸/强化/合成
- 回收产出：银两/少量灵石，作为兜底

保底与赛季化（可选）：
- 天品保底：同一副本/同一Boss累计掉落 N 次必出天品（保底计数写在玩家侧，不写在物品表）
- 词条保底：重铸若干次后提升“目标词条出现权重”（避免极端脸黑）

### 7.10 数据字段补充建议（在现有表基础上最少新增）

在尽量少增字段的前提下，若要把装备系统落库更顺畅，建议新增或明确以下字段用途：

- `item_def`
  - `implicit_attrs`（jsonb，可选）：固有属性（不随机）
  - `affix_count_min` / `affix_count_max`（int，可选）：覆盖默认品质规则（用于特殊装备）
  - 现有 `affix_pool_id`：词条池ID（必用）
  - 现有 `level` + `equip_req_realm`：作为装备境界等级来源（决定词条Tier上限）
- `item_instance`
  - `identified`（boolean，可选）：是否已鉴定
  - 现有 `affixes`：随机词条结果（最终Roll）
  - 现有 `random_seed`：用于复现生成（可选）

### 7.11 套装系统（Set）策划（补齐）

目标：提供“刷套装/配套装”的长期追求，让装备玩法从“单件词条”扩展到“组合构筑”。

#### 7.11.1 核心概念

- 套装（Set）：一组拥有同一 `set_id` 的装备集合（例如“寒月套装”）。
- 套装件（Piece）：某个 `equip_slot` 的具体装备定义（`item_def`），通过 `set_id` 归属某套装。
- 套装加成（Set Bonus）：当玩家穿戴同一套装的件数达到阈值时触发的效果（例如 2/4/6 件效果）。

#### 7.11.2 触发规则（回合制友好）

- 统计范围：只统计 `location=equipped` 的装备实例。
- 计数逻辑：同一 `set_id` 的“不同槽位件”计入件数；重复槽位不叠加计数。
- 生效时机：战斗开始时结算一次；战斗中换装则在下一次进入战斗时重新结算（避免回合中频繁结算）。
- 套装效果表达：复用 `effect_defs` 的结构（属性加成/回合触发/命中触发/受击触发/释放技能触发等），以保证与装备特效、道具效果同一套实现。

#### 7.11.3 套装阈值建议（可配置）

为了适配不同套装件数（2件套/4件套/6件套/8件套），建议阈值采用配置，不写死：

- 常见阈值：2 / 4 / 6（或 2 / 3 / 5）
- 8槽位体系下的推荐：2 / 4 / 6 / 8（高阶套装可启用 8 件）

#### 7.11.4 套装加成类型（完整覆盖）

1) 数值类（稳定收益）：
- 基础属性：物攻/法攻/防御/气血/速度/命中/闪避等
- 资源类：回合回气、技能消耗降低、护盾效率等

2) 机制类（构筑核心）：
- 回合开始/结束触发：叠层、回气、护盾、净化
- 命中/受击触发：追加伤害、反击、施加控制/减益
- 技能联动：当施放某类技能时触发（例如“攻击类功法命中后，有概率追加1次弱化效果”）

3) 限定类（平衡约束）：
- 触发次数：每回合/每战斗触发上限
- 冷却回合：触发后进入冷却
- 触发门槛：需要目标处于某状态/自身处于某状态

#### 7.11.5 与暗黑随机词条的关系

- 套装加成与随机词条独立：随机词条决定“单件差异”，套装加成决定“组合追求”。
- 套装件本身仍遵循品质→词条条数规则（黄/玄/地/天）。
- 高阶套装可以在 `item_def` 上配置更强的 `affix_pool_id`（更偏向该套装流派），但不强制固定词条，保持刷装乐趣。

#### 7.11.6 产出与保底（刷装体验）

- 产出来源：
  - Boss/秘境：掉落某套装的若干槽位（按 `equip_slot` 划分掉落池）
  - 合成/兑换：用分解材料兑换“指定槽位随机件”（避免卡死）
- 保底建议：
  - 槽位保底：同一副本连续 N 次未出该槽位套装件则提升权重
  - 套装收集保底：集齐若干“套装残片”可兑换缺失槽位（随机词条仍随机）

#### 7.11.7 数据结构建议（不强制新增列，优先扩展表）

现有字段：
- `item_def.set_id`：套装ID（用于归属）

建议扩展表（更利于策划配置与运营热更）：

`item_set`（套装定义）
- id：套装ID（中文备注：套装ID）
- name：套装名称（中文备注：套装名称）
- quality_rank：套装档位（中文备注：套装强度档位）
- min_realm：最低境界门槛（中文备注：套装最低适用境界）
- desc：套装说明（中文备注：套装说明）
- enabled/version：启用与版本（中文备注：运营开关与版本）

`item_set_piece`（套装件映射）
- set_id：套装ID（中文备注：套装ID）
- equip_slot：槽位（中文备注：装备槽位）
- item_def_id：物品定义ID（中文备注：套装件物品ID）
- piece_key：套装件键（中文备注：套装件唯一键，例如 head/clothes）

`item_set_bonus`（套装加成）
- set_id：套装ID（中文备注：套装ID）
- piece_count：触发件数（中文备注：触发件数阈值）
- effect_defs(jsonb)：效果定义（中文备注：套装加成效果定义，结构与 item_def.effect_defs 统一）
- priority：优先级（中文备注：与其他效果叠加顺序）
