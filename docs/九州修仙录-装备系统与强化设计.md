# 九州修仙录 - 装备系统与强化设计

> 版本：v0.1  
> 日期：2026-02-03  
> 目标：梳理当前装备系统的真实实现流程，并给出可落地的“装备强化”策划与技术接入方案。

## 1. 当前装备系统（已实现）

### 1.1 核心分层

- **物品定义（静态）**：`item_def`  
  用于：装备模板、基础属性、品质、装备槽位、词条池、套装归属等。
- **物品实例（动态）**：`item_instance`  
  用于：玩家背包/仓库/已穿戴的具体装备实例；包含随机词条、强化等级、是否鉴定、绑定状态等。

### 1.2 主要数据表

- `item_def`：装备模板数据（`category='equipment'`）
- `item_instance`：装备实例数据（玩家拥有的那一件）
- `affix_pool`：随机词条池（抽取规则 + 词条定义）
- `item_set` / `item_set_piece` / `item_set_bonus`：套装与套装加成

> 表结构定义与中文备注见：`server/src/models/itemTables.ts`

### 1.3 种子数据（策划入口）

- 装备模板：`server/src/data/seeds/equipment_def.json`（落库到 `item_def`）
- 通用物品：`server/src/data/seeds/item_def.json`（含强化材料/强化符/保护符等）
- 词条池：`server/src/data/seeds/affix_pool.json`（落库到 `affix_pool`）
- 套装：`server/src/data/seeds/item_set.json`（落库到 `item_set*`）

种子加载逻辑：`server/src/services/seedService.ts`

## 2. 装备生成（掉落/发放装备时用）

实现位置：`server/src/services/equipmentService.ts`

### 2.1 生成流程（真实实现）

1. **读取装备模板**（`item_def`）  
   - 入口：`getEquipmentDef(itemDefId)`
2. **抽取品质**（支持指定或权重随机）  
   - 入口：`rollQuality(...)`
   - 支持 `quality_min/quality_max` 限定区间；支持福源 `fuyuan` 对高品质权重加成
3. **基础属性缩放**（按品质阶梯缩放）  
   - 系数：`黄=1，玄=1.2，地=1.45，天=1.75`
   - 规则：按“实例品质/模板品质”比值缩放 `base_attrs`
4. **抽取随机词条**（按词条池 + 品质决定条数）  
   - 入口：`getAffixPool(poolId)` → `rollAffixes(...)` → `rollAffixValue(...)`
   - 词条 tier 受 `realmRank` 限制（`tier.realm_rank_min <= realmRank`）
5. **创建实例写入 `item_instance`**  
   - 入口：`createEquipmentInstance(...)` / `createEquipmentInstanceTx(...)`
   - 会写入：`quality/quality_rank/random_seed/affixes/identified/location/location_slot/...`

### 2.2 重要约束

- 生成的是“装备实例”，每件都是 `qty=1`，并具备可复现的 `random_seed`
- 当前属性只在两处参与数值结算：
  - `base_attrs`：基础属性（会做品质缩放）
  - `affixes`：随机词条（目前只对 `apply_type='flat'` 生效）

## 3. 穿戴/卸下与属性结算（已实现）

实现位置：`server/src/services/inventoryService.ts`

### 3.1 穿戴流程（equip）

入口：`equipItem(characterId, userId, itemInstanceId)`

- 加锁与校验：锁定 `item_instance` 行，校验是否锁定、是否为装备、是否已穿戴、位置是否合法（背包/仓库）
- 计算属性增量：
  - 新装备：`getEquipmentAttrDeltaByInstanceIdTx(...)`
  - 若槽位已有装备，会先卸下旧装备并扣除旧装备属性
- 写入穿戴状态：
  - `location='equipped'`
  - `equipped_slot=item_def.equip_slot`
  - 若装备定义是 `bind_type='equip'` 且实例未绑定，则穿戴时写入绑定归属（用户/角色）
- 套装加成差分：
  - 穿戴前：`getEquippedSetBonusDeltaTx(...)`
  - 穿戴后：`getEquippedSetBonusDeltaTx(...)`
  - 应用：`after - before` 的差分写入角色属性

### 3.2 卸下流程（unequip）

入口：`unequipItem(characterId, itemInstanceId, { targetLocation })`

- 校验必须处于 `location='equipped'`，并且未锁定
- 计算装备属性增量并取反扣除
- 将装备移动到背包/仓库空格
- 同样会做套装加成差分结算

### 3.3 当前“装备属性”结算规则

`getEquipmentAttrDeltaByInstanceIdTx(...)` 的关键规则：

- 只处理 `item_def.base_attrs`（按实例品质做倍率缩放）
- 只处理随机词条里 `apply_type='flat'` 的 `attr_key/value`
- 套装只处理 `effect_type='buff'` 且 `trigger='equip'` 且 `target='self'` 且 `apply_type='flat'`

> 现状影响：`strengthen_level`、`refine_level`、宝石镶嵌等字段已落库，但目前不会改变角色属性。

## 4. 强化系统现状（数据已准备，但未接入属性）

### 4.1 数据字段（已存在）

- `item_instance.strengthen_level`：强化等级（默认 0）
- `item_instance.refine_level`：精炼等级（默认 0）

定义位置：`server/src/models/itemTables.ts`

### 4.2 强化道具（已在种子数据中配置）

位于：`server/src/data/seeds/item_def.json`

- 强化材料
  - `enhance-001`：淬灵石（强化 1-10）
  - `enhance-002`：蕴灵石（强化 10-15）
- 提升成功率（建议在“强化”操作里作为可选消耗）
  - `enhance-003`：强化符·黄（`success_rate_bonus: 1000`，即 +10%）
  - `enhance-004`：强化符·玄（`success_rate_bonus: 2000`，即 +20%）
- 失败保护（建议在“强化”操作里作为可选消耗）
  - `enhance-005`：保护符（`protect_downgrade: true`）

> 注意：当前“使用物品”逻辑只支持 `trigger='use'`，强化相关道具的 `trigger='enhance'` 需要由强化接口来消费与解析。

## 5. 装备强化策划（建议方案，可直接落地实现）

### 5.1 设计目标

- 给装备提供持续成长线（银两消耗 + 材料消耗），建立中后期资源出口
- 不破坏“刷词条/换装备”的核心循环：强化提升主要作用在基础属性上，词条仍然是差异化核心

### 5.2 强化对象与限制

- 仅 `category='equipment'` 的实例可强化
- `qty` 必须为 1
- 建议限制：锁定物品不可强化（复用 `locked`）
- 建议限制：未鉴定装备不可强化（复用 `identified`，避免“未鉴定刷胚子”）
- 建议允许：装备在背包、仓库、已穿戴三种状态都可强化（便于体验），但实现时要保证穿戴状态下属性会正确重算（见 5.6）

### 5.3 强化等级上限

建议先按当前材料分段直接定到 **+15**：

- 强化等级范围：`0 ~ 15`
- 消耗材料分段：
  - `1 ~ 10`：淬灵石（`enhance-001`）
  - `11 ~ 15`：蕴灵石（`enhance-002`）

后续可扩展：按品质决定上限（例如黄10/玄12/地15/天18），但第一版建议统一 +15 降低复杂度。

### 5.4 成功率与失败惩罚（建议一版）

成功率（不含道具加成）建议表：

| 目标等级 | 成功率 |
|---:|---:|
| +1 ~ +5 | 100% |
| +6 | 80% |
| +7 | 70% |
| +8 | 60% |
| +9 | 50% |
| +10 | 40% |
| +11 | 35% |
| +12 | 30% |
| +13 | 25% |
| +14 | 20% |
| +15 | 15% |

失败惩罚建议：

- +1 ~ +7：失败不降级（只消耗材料与货币）
- +8 ~ +15：失败 **默认降 1 级**；若消耗了 `保护符`（`enhance-005`）则 **不降级**

成功率道具加成（可选消耗）：

- 强化符·黄：+10%（+1000 万分比）
- 强化符·玄：+20%（+2000 万分比）

### 5.5 强化的属性收益（建议一版）

建议强化仅影响“基础属性（base_attrs）”，不影响随机词条；收益曲线用“百分比乘算”便于统一处理：

- 对“用于角色面板的属性键”（`inventoryService` 里的 `allowedCharacterAttrKeys`）生效
- 强化加成倍率：
  - `强化倍率 = 1 + strengthen_level × 0.03`
  - 即每级 +3% 的基础属性（+10 ≈ +30%，+15 ≈ +45%）

按该规则，装备综合强度主要仍由品质与词条决定，强化提供稳定但不会过度爆炸的成长。

> 后续可用 `item_def.growth_attrs` 支持“指定装备的个性化强化曲线”（例如武器每级 +4%，防具每级 +2%），但第一版建议先统一规则。

### 5.6 技术接入点（实现路径）

要让强化“真实影响属性”，建议改造点如下（按现有架构最少改动）：

1. 在 `getEquipmentAttrDeltaByInstanceIdTx(...)` 中加入强化系数  
   - 读取 `ii.strengthen_level`
   - 在当前 `base_attrs` 做完品质缩放后，再按强化倍率二次缩放
2. 若允许“穿戴中强化”，强化成功/失败后需要即时刷新角色属性  
   - 方案 A：强化接口内部直接调用 `applyCharacterAttrDeltaTx` 做差分（推荐，性能好）  
   - 方案 B：强化后强制“卸下再穿戴”触发重算（简单但体验差）
3. 新增强化接口（建议放在 `inventoryRoutes.ts` 下）  
   - `POST /inventory/enhance`：入参包含 `itemInstanceId`、可选的“强化符/保护符实例ID”、是否使用灵石/银两等
4. 强化事务内处理三类扣除：
   - 扣材料（淬灵石/蕴灵石）
   - 扣货币（银两/灵石）
   - 扣可选道具（强化符/保护符）

### 5.7 强化消耗建议（第一版）

货币建议以 **银两为主、灵石为辅**，并与装备等级/品质挂钩：

- 银两消耗：`silver_cost = (item_def.level + 5) × 20 × (targetLevel)`
- 灵石消耗（可选）：`spirit_cost = floor(targetLevel / 5)`（每 5 级 1 灵石）
- 材料消耗：
  - `1~10`：每次 1 个淬灵石
  - `11~15`：每次 1 个蕴灵石

数值可在上线后按经济回收速度调整；公式只作为第一版默认。

## 6. 下一步建议（与现有代码最贴合）

- 先实现“强化影响属性”与“强化接口 + 材料/货币扣除 + 成功率/保护符”
- 精炼（`refine_level`）与宝石镶嵌（`socketed_gems`）暂不做，等装备系统闭环后再扩展
